## 1. 버전 관리란?
- 파일이나 코드의 변경 이력을 체계적으로 기록하고 관리하는 것

### 1-1. 버전 관리가 필요한 이유
- **복구**: 실수로 코드를 지웠거나 버그가 발생했을 때, 과거의 안정적인 상태로 되돌릴 수 있다.
- **협업**: 여러 명이 동시에 같은 파일을 수정할 때 발생하는 충돌을 체계적으로 해결할 수 있다.
- **이력** 추적: 누가, 언제, 어떤 이유로 코드를 수정했는지 상세히 파악할 수 있다.
- **실험**: 원본 코드에 영향을 주지 않고 새로운 기능을 테스트해 볼 수 있다.

## 2. 필수 용어 정리

|용어|정의 및 설명|
|--|--|
|Repository|프로젝트의 모든 변경 이력이 저장되는 저장소|
|Commit|특정 시점의 변경 사항을 기록하는 행위 (= 하나의 버전)|
|Branch|독립적으로 작업을 진행하기 위해 원본에서 분리된 작업 공간|
|HEAD|현재 내가 작업 중인 위치를 가리키는 포인터|

## 3. Git의 영역
1. **Working Directory** : 실제로 파일을 수정하고 있는 내 컴퓨터의 작업 공간
2. **Staging Area** : 커밋하기 전, 저장소에 기록할 파일들을 잠시 추려두는 준비 영역
3. **Repository** : 최종적으로 버전이 기록되는 저장 공간

## 4. Git 설치 및 기본 사용법
### 4-1. 설치 및 초기설정
```bash
# Git 설치 (macOS)
brew install git

# 설치 확인
git --version

# (전역) 사용자 이름, 이메일 설정 
git config --global user.name "unz2"
git config --global user.email "unz2@example.com"

# (로컬) 사용자 이름 설정, 이메일 설정 
git config user.name "unz2"
git config user.email "unz2@example.com"

# 설정 확인
git config --list

```

### 4-2. 저장소 생성 및 상태 확인
```bash
# 1. 저장소 초기화
# .git 폴더 생성, 현재 폴더를 기준으로 버전 관리 시작
git init

# 2. 현재 저장소 상태 확인
# 수정된 파일, 스테이징 여부, 커밋 가능한 변경 사항 확인
git status

```

### 4-3. 저장 및 커밋
```bash
# 3. 파일을 Staging Area로 올리기 (특정 파일)
git add 파일명.py

# 모든 수정 사항을 올리기
git add .

# 4. 커밋하기 (Staging Area에 있는 변경 사항을 하나의 버전으로 기록)
git commit -m "커밋 메시지"

```

### 4-4. 커밋 메시지 작성 규칙
- 좋은 커밋 메시지는 무엇을 했는지 한눈에 알 수 있어야 한다.
- 구조 : `Type: Subject`

|Type 종류||
|--|--|
|`feat`|새로운 기능 추가|
|`fix`|버그 수정|
|`docs`|문서 수정|
|`style`|코드 포맷팅, 세미콜론 누락 등|
|`refactor`|코드 리팩토링|
|`test`|테스트 코드 추가|

## 5. .gitignore
- 버전 관리에서 제외할 파일을 지정하는 설정 파일
- 환경 변수  파일, 빌드 결과물, 개인 설정 파일 등

```
# .gitignore 파일 예시
node_modules/      # 의존성 폴더 제외
.env               # 보안이 필요한 환경변수 파일 제외
*.log              # 모든 로그 파일 제외
build/             # 빌드 결과물 제외

<주의> 이미 커밋된 파일은 Git이 계속 추적하기 때문에, git rm --cached 파일명
```

## 6. 브랜치(Branch)
- 기존 작업과 분리된 독립적인 작업 흐름
- 브랜치를 통해 한 저장소 안에서 여러 작업을 동시에 병렬로 진행할 수 있다.
- 기능 개발, 실험, 수정 작업을 안전하게 진행할 수 있다.
- 협업과 안정적인 배포를 위해서는 브랜치를 잘 관리하는것이 중요하다.

|주요 브랜치 종류||
|--|--|
|`main`|최종적으로 사용자에게 배포되는 브랜치|
|`dev`|다음 버전을 위한 개발이 진행되는 통합 브랜치|
|`feature/*`|기능을 단위별로 개발하는 임시 브랜치 (예: `feature/ui`)|
|`hotfix/*`|출시된 버전에 긴급한 버그가 생겼을 때 빠르게 수정하기 위한 브랜치|

### 6-1. 브랜치 명령어
```bash
# 브랜치 목록 확인
git branch

# 브랜치 생성
git branch [브랜치명]

# 브랜치 이동
git switch [브랜치명]

# 브랜치 생성과 동시에 이동
git checkout -b [브랜치명]

# 브랜치 삭제
git branch -d [브랜치명]

```

## 7. Git Merge
```bash
# 다른 브랜치의 작업 내용을 현재 브랜치로 병합하는 명령어
git merge [브랜치명]
```

### 7-1. Fast-forward vs 3-way Merge
- **Fast-forward** : 브랜치가 직선으로 이어질 때, 단순히 커밋 포인터만 이동
- **3-way Merge** : 브랜치가 갈라졌다가 다시 합쳐질 떄, 병합 커밋이 생성됨

### 7-2. Merge Conflict 해결
- 같은 파일의 같은 부분을 서로 다르게 수정했을 떄 발생한다.
- Git은 자동 병합을 시도하고, 판단이 필요한 경우만 충돌로 알려준다.

1. `git status`로 충돌 파일 확인
2. 에디터에서 `<<<<,` `====,` `>>>>` 표시를 보고 코드 수정
3. `git add` 후 `git commit`으로 마무리

### Merge Conflict 해결 실습
```bash
# Git 레포지토리 초기화
git init

echo "hello" > test.txt
git add .
git commit -m "first commit"

# feature 브랜치 생성
git branch feature
git switch feature

# 같은 파일 수정
echo "hello from feature" > test.txt
git add .
git commit -m "update from feature"

# main에서 다시 수정
git switch main
echo "hello from main" > test.txt
git commit -am "update from main"

# 병합 시도
git merge feature

# 충돌 확인
cat test.txt

# 파일 수정
vi test.txt
git commit -am "resolve merge conflict"
git log --oneline --graph

```

## 8. GitHub
- Git 저장소를 호스팅하는 서비스
- 계정을 만든 뒤 새 Repository를 생성하면 원격 저장소 주소(URL)가 발급

```bash
git remote add origin [원격 저장소 주소(URL)]
git remote -v
```

### 8-1. Git / GitHub
- **Git**: 내 컴퓨터에서 작동하는 버전 관리 '도구'
- **GitHub**: Git 저장소를 호스팅해주는 '클라우드 서비스'

### 8-2. git push / git pull
- 로컬과 원격 저장소를 동기화하는 명령어
- `git push`: 로컬 커밋을 원격 저장소로 업로드
- `git pull`: 원격 변경 사항을 내려받아 로컬에 반영

```bash
# 로컬 커밋을 원격 저장소로 업로드
git push -u origin main

# 원격 저장소의 변경 사항 다운로드
git pull
```

### 8-3. Clone / Fork
- **Clone**
    - 특정 저장소를 내 컴퓨터로 그대로 복제해 오는 것 
    - 같은 저장소에 push 권한이 있는 경우 사용
    - 팀 내부 협업에 주로 사용
- **Fork**
    - 타인의 저장소를 내 GitHub 계정으로 '복제'해 가는 것
    - 원본 저장소에 직접 push 권한이 없는 경우 사용
    - 오픈 소스 기여 시 주로 사용

### 8-4. 협업 프로세스 도구
- **Pull Request (PR)**
    - 내가 작업한 코드, 변경사항을 메인 코드에 합쳐달라고 요청하는 것
    - 변경 내용 설명
    - 리뷰 요청
    - 병합 전 검증 단계
- **Code Review**
    - PR 단계에서 팀원들이 코드를 검토하고 의견을 나누는 과정
    - 코드를 병합하기 전 함께 확인하는 과정 (버그 예방, 코드 품질 유지, 팀 규칙 공유)
    - 코드 중심으로 리뷰
    - 이유를 남긴 코멘트
    - 승인 후 병합
- **Issue**
    - 프로젝트의 버그, 건의 사항, 할 일 목록 등을 기록하고 관리하는 기능
    - PR과 Issue를 연결하여 직접 관리
